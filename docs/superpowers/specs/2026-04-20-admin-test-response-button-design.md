# 管理页固定 hello 测试请求按钮设计

## 背景

当前管理页已经支持配置账号、查看代理 `apikey`、新增和删除配置项，但缺少一个可以直接验证代理链路是否正常的页面内测试入口。现有 README 里提供了 `curl /v1/responses` 的手工测试方式，本次改动把这个验证动作下沉到管理页，减少手工复制和命令行操作。

## 目标

- 在管理页增加一个“测试请求”按钮。
- 点击后固定向当前实例的 `/v1/responses` 发送一条内容为 `hello` 的请求。
- 页面内展示请求结果，便于快速确认代理链路是否可用。

## 非目标

- 不增加自定义输入框，不支持修改测试文本。
- 不新增独立的后端测试接口。
- 不调整现有配置管理和额度刷新逻辑。

## 方案概述

采用前端直接调用当前服务 `/v1/responses` 的方式，而不是新增 `/admin/api/test-response` 中转接口。

原因如下：

- 可以直接验证真实入口链路：管理页 -> 本地代理 `/v1/responses` -> 上游服务。
- 不引入额外的后端转发逻辑，代码和维护面最小。
- 管理页已经能拿到 `apikeys` 列表，可复用现有配置完成鉴权。

## 页面与交互

按钮放在管理页顶部工具栏，和“刷新”按钮并列，名称为“测试请求”。

点击按钮后的交互行为：

- 进入请求中状态时禁用按钮，避免重复点击。
- 成功时在现有消息区域显示成功提示，并附带一段简短响应摘要。
- 失败时在现有消息区域显示错误信息。

不新增独立弹窗或结果面板，继续复用当前页面已有的消息展示区域。

## 请求设计

请求地址：

- `POST /v1/responses`

请求头：

- `Content-Type: application/json`
- 当当前配置存在代理 `apikeys` 时，附带第一个 `apikey` 到 `Authorization: Bearer <apikey>`

请求体：

```json
{
  "model": "<claude_code.model 或默认 gpt-5.4>",
  "input": [
    {
      "type": "message",
      "role": "user",
      "content": [
        {
          "type": "input_text",
          "text": "hello"
        }
      ]
    }
  ]
}
```

模型选择规则：

- 优先使用管理页配置快照中的 `claude_code.model`
- 如果未配置，则回退到 `gpt-5.4`

## 响应展示

成功后优先从返回 JSON 中提取可读文本摘要，显示在消息区域中。若无法稳定提取文本，则退化为展示“测试请求成功”以及响应 JSON 的简短序列化结果。

考虑到当前页面消息区以单段文本为主，本次实现只展示一段简短结果，不扩展为复杂的多区域渲染。

## 错误处理

- `401`：优先提示可能是代理 `apikey` 未携带或已失效。
- `503`：提示当前没有可用配置，需要先添加账号。
- 其他非 `2xx`：展示后端返回的 `details`、`message` 或 `error`。
- 网络异常或解析失败：展示通用失败信息，并保留原始错误文本。

## 测试策略

新增前端页面行为测试暂不适合当前项目结构，本次以 Node 原生测试补充一个纯函数级别的结果提取测试，或对现有管理页静态脚本进行轻量验证；同时进行手工验证：

- 管理页能看到“测试请求”按钮。
- 有可用配置时点击按钮，会向 `/v1/responses` 发送固定 `hello`。
- 配置了代理 `apikey` 时，请求会自动带上第一个 `apikey`。
- 成功和失败场景都能在页面消息区看到明确反馈。

## 实施边界

本次变更预计只涉及：

- 管理页静态文件 `public/config-admin.html`
- 如有必要，新增一个小型测试文件用于验证响应摘要提取逻辑
